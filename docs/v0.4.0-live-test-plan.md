# CoconutClaw v0.4.0 Live 测试计划

## 目标

在真实 Slack / Telegram 环境中验证以下能力是否可用且符合预期：

- Slack 工作区集成
- Slack Thread / Session 隔离
- 跨 Session 并行多任务
- 同一 Session 内 FIFO 顺序执行
- `/tasks`
- `/cancel`
- `/cancel <task-id>`
- `/fresh`
- 审批流（`SEND_APPROVAL`）
- 基础管理员权限控制
- 崩溃隔离 / 重启恢复
- Telegram 多 Chat / Session 支持
- Telegram 回归安全

---

## 0. 测试准备

### 建议环境
- 使用测试 Slack 工作区 / 测试频道，不要直接在生产频道验证。
- 建议同时准备：
  - 1 个管理员账号
  - 1 个非管理员账号

### 关键配置
确认以下配置可用：

- `SLACK_BOT_TOKEN`
- `SLACK_APP_TOKEN`（如果使用 Socket Mode）
- `SLACK_SIGNING_SECRET`（如果使用 webhook 交互）
- `SLACK_CHANNEL_ID`
- `SLACK_ADMIN_USER_IDS`

如需验证 Telegram 多 chat，额外确认：

- `TELEGRAM_BOT_TOKEN`
- `TELEGRAM_CHAT_ID`（保留单默认 chat 兼容）
- `TELEGRAM_CHAT_IDS`（可选，逗号分隔 allowlist；用于多 chat 接入验证）

### 建议留证
每个用例建议记录：

- Slack thread 链接
- 截图
- 运行日志片段
- PASS / FAIL
- 备注

## 1. 启动与连通性验证

### 步骤
1. 启动 CoconutClaw 的 Slack 运行模式。
2. 确认运行时正常连接，无启动报错。
3. 在 Slack 中发送一条普通消息给机器人。

### 预期
- 机器人收到消息
- 出现进度消息
- 最终回复返回到正确位置
- 无崩溃
- 无重复回复

---

## 2. Slack 基础命令验证

### 2.1 `/tasks`
#### 步骤
1. 在当前没有任务运行时执行 `/tasks`。

#### 预期
- 返回 “No active tasks” 或等价空任务列表

### 2.2 `/fresh`
#### 步骤
1. 在一个 Slack thread 中先发一条普通消息。
2. 再发一条依赖前文上下文的问题。
3. 执行 `/fresh`。
4. 再发一条原本依赖前文的问题。

#### 预期
- 返回上下文已清空的确认消息
- 后续对话按全新上下文处理
- 只影响当前 thread / session，不影响其他 thread

### 2.3 `/cancel`
#### 步骤
1. 在一个 thread 中发起一个明显会运行较久的请求。
2. 任务运行中执行 `/cancel`。

#### 预期
- 返回取消请求已提交
- 任务停止或进入 `cancel_requested` / `cancelled`
- 当前 thread 后续仍可正常继续使用

### 2.4 `/cancel <task-id>`
#### 步骤
1. 启动多个任务。
2. 执行 `/tasks`。
3. 记录其中一个任务 ID。
4. 执行 `/cancel <task-id>`。

#### 预期
- 只取消指定任务
- 其他任务继续执行

### 2.5 `/schedules`
#### 步骤
1. 执行 `/schedules`。

#### 预期
- 正常列出计划任务
- 无崩溃、无格式错乱

---

## 3. Thread 隔离验证

### 步骤
1. 在同一个 Slack channel 中创建 **Thread A**。
2. 在 Thread A 里说：`记住口令 ALPHA`。
3. 创建 **Thread B**。
4. 在 Thread B 里问：`我的口令是什么？`
5. 回到 Thread A 问：`我的口令是什么？`

### 预期
- Thread B 不应继承 Thread A 的上下文
- Thread A 应能保留 ALPHA
- 不应出现跨 thread 上下文污染

---

## 4. 跨 Session 并行多任务验证

### 步骤
1. 在 **Thread A** 发起一个长任务。
2. 立即在 **Thread B** 发起另一个长任务。
3. 如果方便，也可以同时从 Telegram 或第三个 Slack thread 发起任务。
4. 观察 Slack 进度消息和运行日志。

### 预期
- 多个任务能并行推进
- 彼此不互相阻塞
- 输出都落在各自正确的 thread / channel 中

---

## 5. 同一 Session 内 FIFO 验证

### 步骤
1. 在同一个 Slack thread 内，连续快速发送：
   - 消息 A：长任务
   - 消息 B：第二个请求
2. 观察执行顺序。

### 预期
- 消息 B 不应抢到消息 A 前面执行
- 回复顺序应与提交顺序一致
- 不应出现上下文 race condition

---

## 6. `/tasks` 在并发状态下的准确性

### 步骤
1. 在不同 thread 中启动 2~3 个任务。
2. 在任务运行中执行 `/tasks`。

### 预期
- 可以看到多个任务
- 任务 ID 可区分
- 状态合理，例如：
  - `queued`
  - `running`
  - `awaiting_approval`
  - `cancel_requested`
- 能看出任务归属的 session / thread

## 7. 审批流验证

### 7.1 管理员批准
#### 步骤
1. 触发一个会产出 `SEND_APPROVAL` 的动作。
2. 等待 Slack 显示 Approve / Reject 按钮。
3. 用管理员账号点击 **Approve**。

#### 预期
- 审批消息正常出现
- 按钮点击有效
- 任务恢复执行
- 最终输出成功返回
- `/tasks` 状态发生正确变化

### 7.2 管理员拒绝
#### 步骤
1. 再次触发一个审批。
2. 用管理员账号点击 **Reject**。

#### 预期
- 任务被干净地停止
- 状态进入终态
- 不应继续偷偷执行

### 7.3 非管理员尝试批准
#### 步骤
1. 触发审批。
2. 用非管理员账号点击 **Approve**。

#### 预期
- 操作被拒绝
- 任务不恢复执行
- 有明确未授权提示

## 9. 崩溃隔离验证

### 步骤
1. 在 Thread A 启动任务 A。
2. 在 Thread B 启动任务 B。
3. 让任务 A 尽可能触发失败。
4. 观察任务 B 和主循环状态。

### 预期
- 任务 A 失败
- 任务 B 继续运行并可完成
- 主循环不崩溃
- 系统仍能接受新请求

---

## 10. 重启 / Stale Task 恢复验证

### 步骤
1. 启动一个长任务。
2. 在任务完成前强制杀掉进程。
3. 重启 CoconutClaw。
4. 执行 `/tasks`。
5. 查看日志与 Slack 表现。

### 预期
- 旧的 in-flight 任务被安全标记为 failed / expired
- 系统能正常恢复接收新任务
- 不会产生重复回放 spam

## 12. Telegram 多 Chat / 并发验证

### 12.1 多 Chat 接入
#### 步骤
1. 配置：
   - `TELEGRAM_CHAT_ID=<chat-a>`
   - `TELEGRAM_CHAT_IDS=<chat-b>` 或 `TELEGRAM_CHAT_IDS=<chat-a>,<chat-b>`
2. 从 Chat A 发送普通消息。
3. 从 Chat B 发送普通消息。

#### 预期
- 两个 chat 都能收到响应
- 两个 chat 都被视为独立 session
- 不会出现 “chat_id_mismatch” 误拦截

### 12.2 多 Chat 上下文隔离
#### 步骤
1. 在 Chat A 发送：`记住口令 ALPHA`
2. 在 Chat B 发送：`我的口令是什么？`
3. 回到 Chat A 再问：`我的口令是什么？`

#### 预期
- Chat B 不应继承 Chat A 的上下文
- Chat A 仍能取回 ALPHA

### 12.3 多 Chat 并行
#### 步骤
1. 在 Chat A 发起一个长任务。
2. 立即在 Chat B 发起另一个长任务。
3. 观察两边进度消息与日志。

#### 预期
- 两个 Telegram chat 可并行推进
- 互不阻塞
- 输出回到各自 chat

### 12.4 同 Chat FIFO
#### 步骤
1. 在同一个 Telegram chat 快速连续发送：
   - 消息 A：长任务
   - 消息 B：第二个请求
2. 观察回复顺序。

#### 预期
- 消息 B 不应早于消息 A 执行完成
- 回复顺序与提交顺序一致
- 不出现上下文 race condition

### 12.5 Chat 定向取消
#### 步骤
1. 在 Chat A 启动长任务。
2. 在 Chat B 启动另一个长任务。
3. 只在 Chat B 执行 `/cancel`。
4. 查看 `/tasks` 和两边实际执行结果。

#### 预期
- 只取消 Chat B 当前 session 的任务
- Chat A 任务继续运行
- `/cancel` 回复返回到发起取消的那个 chat
